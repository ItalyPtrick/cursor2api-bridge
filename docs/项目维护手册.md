# 项目维护手册

这份文档是给项目维护者自己的。

它的目标不是介绍软件怎么用，而是告诉你：

- 更新完功能后要做什么
- 怎么重新打包
- 怎么修改文档
- 怎么发布 Release
- 怎么确认用户的软件内更新是真的可用

## 2. 日常改代码后的基本顺序

每次你改完功能后，推荐按这个顺序走：

1. 先确认代码改动已经保存
2. 运行测试
3. 重新打包
4. 本地打开打包后的软件做冒烟测试
5. 如果功能涉及更新链路，再生成正式 Release 资产
6. 最后才 push 到 GitHub 和发布 Release

最常用的命令都在项目根目录执行：

```powershell
cd C:\GIT-TEST\repos\cursor2api-desktop
npm test
npm run dist:dir
```

如果你要直接生成正式发布资产，用：

```powershell
npm run release:assets
```

## 3. 重新打包怎么做

### 3.1 只想打一个目录版做本地测试

在项目根目录运行：

```powershell
npm run dist:dir
```

这个命令会：

1. 清理旧的构建产物
2. 跑测试
3. 编译 TypeScript
4. 准备上游、运行时、图标、版本文件
5. 生成 `release/win-unpacked/`

打包完成后，你主要会用到：

- `release/win-unpacked/Cursor2API Bridge.exe`

### 3.2 想生成正式发布资产

在项目根目录运行：

```powershell
npm run release:assets
```

它会：

1. 先跑 `npm run dist:dir`
2. 再把 `release/win-unpacked/` 打成正式 zip
3. 再生成 zip 对应的 sha256

生成结果在：

```text
release/github-assets/
```

你要关注的是：

- `cursor2api-bridge-portable-win-x64.zip`
- `cursor2api-bridge-portable-win-x64.zip.sha256`

### 3.3 想额外生成单文件 portable exe

在项目根目录运行：

```powershell
npm run dist:portable
```

注意：

- `portable.exe` 只是附加下载
- 软件内更新不用它
- 软件内更新只认 zip 和 sha256

如果你改了下面这些内容，就应该一起改文档：

- 默认端口
- 更新流程
- 资产名称
- 页面结构
- 隐私与日志行为
- 安装方式
- 发布方式

至少检查这些信息有没有保持一致：

- 默认端口是不是仍然是 `3011`
- 资产名是不是仍然是：
  - `cursor2api-bridge-portable-win-x64.zip`
  - `cursor2api-bridge-portable-win-x64.zip.sha256`
- 日志目录是不是仍然是 `data/logs/`
- 新手文档里是不是仍然明确写了“不需要安装 Python / Node.js / Git”

## 5. 发布 Release 的完整流程

### 5.1 先确认更新源配置

打开：

```text
app.manifest.json
```

检查：

```json
"updateFeed": {
  "owner": "ItalyPtrick",
  "repo": "cursor2api-bridge",
  "assetName": "cursor2api-bridge-portable-win-x64.zip",
  "checksumName": "cursor2api-bridge-portable-win-x64.zip.sha256"
}
```

如果 `owner/repo` 错了，软件内更新就会指向错误仓库。  
如果是空的，`npm run release:assets` 会直接拒绝继续。

### 5.2 提交并推送源码

在项目根目录运行：

```powershell
cd C:\GIT-TEST\repos\cursor2api-desktop
git add .
git commit -m "Describe this release"
git push
```

### 5.3 生成正式发布资产

在项目根目录运行：

```powershell
npm run release:assets
```

生成结果会在：

```text
release/github-assets/
```

### 5.4 打开 GitHub 网页创建 Release

在 GitHub 仓库页面：

1. 打开 `Releases`
2. 点击 `Draft a new release`
3. 填 tag，例如 `v0.1.3`
4. 填标题
5. 填说明
6. 上传：
   - `cursor2api-bridge-portable-win-x64.zip`
   - `cursor2api-bridge-portable-win-x64.zip.sha256`
7. 点击 `Publish release`

如果你要额外给用户单文件版本，也可以上传 `portable.exe`，但记得说明：

- 这只是附加下载
- 软件内更新不用它

## 6. 怎么确保用户的软件内更新功能是正常的

这一段最重要。

不能只看“Release 发出去了”，要看“旧版本是不是能自己升到新版本”。

### 6.1 必须满足的前提

软件内更新要成立，必须同时满足：

1. `app.manifest.json` 里的 `updateFeed.owner/repo` 正确
2. GitHub Release 页面里有 zip
3. GitHub Release 页面里有 zip 对应的 sha256
4. 两个文件名都必须完全匹配

### 6.2 发布前验证

发版前至少做一次：

1. 本地生成 zip 和 sha256
2. 确认文件名正确
3. 确认旧版本软件还在

### 6.3 发布后验证

发版后按这套走：

1. 打开旧版本软件
2. 进入 `设置`
3. 点击 `检查更新`
4. 确认它能找到新版本
5. 点击 `下载并解压`
6. 点击 `启动新版本`
7. 确认：
   - 新版本启动成功
   - `http://127.0.0.1:3011/health` 正常
   - `data/` 被自动带过去
   - `data/logs/` 仍然在

### 6.4 如果失败了，优先排查哪里

优先看这几项：

- `app.manifest.json` 里的 `owner/repo`
- Release 资产名是否完全匹配
- 是否漏传了 `.sha256`
- 是否只传了 `portable.exe`
- 旧版本是不是其实还在连旧的仓库地址

## 7. 每次发版后的固定动作

建议把下面这套当作固定习惯：

1. 检查 Release 页面资产
2. 手动下载 zip 试解压
3. 用旧版本跑软件内更新
4. 检查 `3011/health`
5. 检查 `data/` 是否迁移
6. 暂时保留旧版本目录，别马上删

更短的核对表在：

- [发布后检查清单.md](/C:/GIT-TEST/repos/cursor2api-desktop/docs/发布后检查清单.md)

## 8. 哪些东西一般不要上传到 GitHub

通常不要把这些目录直接提交到源码仓库：

- `release/`
- `dist/`
- `vendor/`
- `node_modules/`
- `data/`
- `updates/`

源码仓库主要保留：

- `src/`
- `scripts/`
- `docs/`
- `README.md`
- `LICENSE`
- `package.json`
- `app.manifest.json`

## 9. 维护时最容易踩的坑

### 1. 改了功能，但没重新打包

结果就是：

- 本地源码变了
- 发出去的安装包没变

### 2. Release 上传了 exe，但没上传 zip 和 sha256

这样用户也许能手动下载，但软件内更新不成立。

### 3. 改了更新逻辑，但没用旧版本走一遍真升级

这样最容易出现“看起来都对，用户就是升不上去”。

### 4. 改了端口、目录、资产名，却忘了改文档

这样 README、发布指南和实际行为会冲突。

## 10. 最短维护版流程

如果以后你只是想快速发一个版本，最短流程是：

1. 改代码
2. `npm test`
3. `npm run dist:dir`
4. 本地点开 exe 冒烟测试
5. `git add . && git commit && git push`
6. `npm run release:assets`
7. GitHub 发 Release，上传 zip 和 sha256
8. 用旧版本跑一遍软件内更新
