# 发布后检查清单

这份清单是给发布者自己用的。

每次你在 GitHub 上按下 `Publish release` 后，建议按顺序走一遍，不要跳步。

## 一、先检查 Release 页面本身

打开你刚发布的 Release 页面，确认下面这些内容：

- tag 正确，例如 `v0.1.0`
- 标题正确
- 说明文字是你想展示给用户看的版本
- 下面两个文件都在：
  - `cursor2api-bridge-portable-win-x64.zip`
  - `cursor2api-bridge-portable-win-x64.zip.sha256`

如果少了其中任何一个，软件内更新都不完整。

## 二、手动下载检查

从刚发布的 Release 页面手动下载：

- `cursor2api-bridge-portable-win-x64.zip`

确认：

- 文件能正常下载完
- zip 能正常解压
- 解压后能看到 `Cursor2API Bridge.exe`

## 三、用旧版本跑一遍软件内更新

这一步最重要，因为它决定用户的软件内更新功能是不是真的通了。

操作顺序：

1. 打开旧版本软件
2. 进入 `设置`
3. 点击 `检查更新`
4. 确认它能检测到你刚发布的新版本
5. 点击 `下载并解压`
6. 下载完成后点击 `启动新版本`

## 四、确认新版本真的接管成功

新版本启动后，检查：

- 首页版本号已经变成你刚发布的新版本
- 服务能正常启动
- 浏览器打开 `http://127.0.0.1:3011/health` 能正常返回
- 原项目前端还能打开
- 托盘图标和菜单正常

## 五、确认旧数据是否自动带过去

新版本启动后，再检查：

- `data/` 已经被带过去
- 原来的配置还在
- `data/logs/` 里的日志和数据库还在
- 如果你之前已经有统计数据，统计页还能正常看到旧数据

## 六、确认回滚路径还在

不要急着删旧版本目录。

先确认：

- 旧版本目录仍然存在
- 旧版本目录没有被新版本覆盖
- 如果新版本有问题，你还能手动回到旧目录启动旧版本

## 七、如果发现问题，先检查这些地方

如果旧版本检测不到新版本，先检查：

- `app.manifest.json` 的 `updateFeed.owner/repo` 是否写对
- Release 里是否真的上传了 zip 和 sha256
- 文件名是否完全一致：
  - `cursor2api-bridge-portable-win-x64.zip`
  - `cursor2api-bridge-portable-win-x64.zip.sha256`

如果新版本下载后起不来，先检查：

- 新版本目录里的 `data/`
- 软件内“最近标准错误”
- 软件内“最近错误”
- `http://127.0.0.1:3011/health`

## 八、确认没问题后再做什么

如果这次发布确认没问题，你再做这几件事：

1. 把这次 Release 链接发给朋友
2. 告诉朋友优先用软件内更新，或直接下载 zip
3. 保留旧版本目录一段时间，不要立刻删

## 最短版

如果你赶时间，最少也要做这 5 件事：

1. 确认 Release 页面里有 zip 和 sha256
2. 旧版本能检测到新版本
3. 新版本能启动
4. `3011/health` 正常
5. `data/` 成功带过去
